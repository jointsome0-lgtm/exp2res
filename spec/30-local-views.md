## §30. Local Views

§30 specifies the post-CLI local-view shape for selfos embedding. V1 ships the CLI first; view implementation is explicitly deferred, with the mirror HTML view first and the JD-to-resume view second. This deferral does not relax the contracts below when either view is implemented.

1. **Coupling and serving boundary.** A view is served only on a loopback local URL and is embedded only by that URL, configured outside Exp2Res. Exp2Res stores no consumer identity, shell schema, callback, webhook, or consumer-specific protocol. Serving the view is local presentation under §29.2, not egress, and grants no new network authority.
2. **Read and action authority.** View rendering is read-only over selected current derived state. A user action may delegate only to an existing §14 flow with that flow's selectors, workspace lock, confirmation, exit class, and §14.14 result semantics; the view defines no new command, business mutation path, implicit latest selector, consent default, LLM call, or background work. A view-triggered provider call remains foreground and user-initiated under §29.2, and the view may not poll, queue, retry, resume, or complete it independently.
3. **Mirror view.** The mirror view renders the §17 report for one explicitly selected current `AssessmentSnapshot` as local HTML, preserving §16.11 status gates, uncertainty, contradictions, and source/generated-voice boundaries. It never renders a partial claim set to work around a gate. Interactive provenance drill-down from claim to signal to fact to evidence is a named later iteration and is not part of the first mirror view.
4. **JD-to-resume view.** The view accepts owner-pasted vacancy text, writes the exact validated bytes to one owner-only operation-owned temporary file inside the workspace, requires that file to pass §29.4, supplies its path to the existing §14.10 JD-add operation, and removes it on completion under §29.6; a validation or cleanup failure is an operational failure and reports any residual path. The remaining generation, verification, and export work delegates to the same §14.10 flow without adding another business persistence or model contract. A successful export is one outcome; a refusal required by §18 or a §16.11 gate is an equally first-class completed `blocked` outcome under §14.14, with its reason and findings shown rather than replaced by an error page. An operational failure remains an operational failure and may not be relabeled as a semantic refusal.
5. **Gap-question handoff.** The sole outbound question-of-the-day artifact is the selected current assessment view's managed `out/assessment/<view>/gap_questions.md` from §13.12, containing only the still-unanswered `GapQuestion.question` values selected by §17's Questions Worth Answering projection. The shell may discover that file through configuration, but Exp2Res receives no consumer identity or answer callback. An answer returns only through ordinary diary/activity capture or import as a new `RawLog`; it carries no gap ID or link-back token, and §14.7's no-relink rule remains unchanged.

These views add no LLM transmission field, so §29.3 does not widen. They expose no source payload, result, or action beyond the selected §17 projection, the managed gap file, and the same §14 flows available without a view.
